DNA · foundry-assured · spike protótipo — avaliar s-foundry-emit-prototype

Autora em DNA. Emite o nativo.
O runtime carrega o nativo, não o DNA.

O modelo "Terraform" provado no consumidor real: dna emit gera o PromptAgent nativo do agent-framework em build-time — e o runtime deixa de depender do dna-sdk. Contraste honesto com o modelo runtime-compose que o foundry usa hoje.

01 Dois modelos, mesma fonte de autoria

runtime-compose hoje

O DNA viaja no deploy; o prompt é composto no processo, a cada boot.

author.dna/helpdesk — Soul + Guardrails + instruction
↓ COPY .dna/ + dna-sdk na imagem
runtimedna.load_prompts("helpdesk") compõe agora → instructions

dna-sdk é dependência de runtime. Ganha no-deploy dinâmico e overlay por tenant.

build-emit terraform

Só o YAML nativo viaja; a composição é congelada no build.

author.dna/helpdesk — Soul + Guardrails + instruction
↓ dna emit --target agent-framework (build)
runtimeAgentFactory.create_agent_from_yaml_path() → Agent vivo, SEM dna-sdk

dna-sdk sai do runtime. É ferramenta de build, como terraform apply.

Gate .NET/Azure honesto: o PromptAgent emitido não tem bloco model: (modelo unbound no DNA), então o AgentFactory usa um chat client injetado (stub) e constrói 100% offline — nunca resolve Foundry/Azure nem dispara o runtime .NET.

O próprio dna emit reporta o de-para — eixos DNA sem slot no target: composition structure (Soul+Guardrails achatam), tenant overlay, eval-as-contract, model unbound. Build-emit troca dinamismo por runtime enxuto: o artefato é um snapshot.

04 A crítica de empacotamento do dono — resolvida

O gap: artefato de authoring acoplado ao runtime

Hoje o backend leva o scope .dna/ para dentro da imagem — Dockerfile faz COPY .dna/ e o prompts.py resolve o diretório com o frágil Path(__file__).resolve().parents[2] / ".dna" (+ mount Azure Files /mnt/dna).

Build-emit elimina isso na raiz: o runtime consome só o concierge.agent.yaml nativo. Sem .dna/ na imagem, sem parents[2], sem dna-sdk no requirements, sem mount de scope. O dna emit roda no pipeline de build — o que chega em produção é um artefato de primeira classe do próprio agent-framework.

05 Recomendação — híbrido

build-emit default

Personas estáveis, valor = runtime enxuto + imagem limpa + auditabilidade.

  • Agentes de produto core: concierge grounded/ungrounded, triage → retrieve → resolve.
  • Pipeline roda dna emit e versiona os *.agent.yaml.
  • Mata o gap de empacotamento Docker.

runtime-compose onde precisa

Onde no-deploy dinâmico + tenant overlay são o produto.

  • Troca de prompt por tenant sem redeploy; A/B de persona.
  • Hotfix de guardrail em produção via publish de scope (ADR-014 / /mnt/dna).
  • Mantém load_prompts só nesses scopes.
Veredito Emitir no build por padrão (o default vira Terraform), e manter o caminho load_prompts só nos scopes/tenants que precisam de override dinâmico. O DNA continua a fonte de autoria única nos dois casos — muda só quando a composição é resolvida: build-time vs boot-time. É exatamente a promessa Terraform.
