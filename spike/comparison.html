The verdict, up front
Conditional — the runtime count decides

As a prompt-manager for a single agent-framework app, DNA does not justify itself. It earns its keep only as a neutral author-once → emit-to-each-runtime layer.

agent-framework already ships a clean declarative YAML loader that lives in this backend's dependency tree. For one runtime, DNA's Genome/Soul/Guardrail apparatus is more ceremony for the same composed prompt. But the spike proves DNA can transpile its definition into the exact PromptAgent YAML agent-framework consumes (byte-equal), and 2026 research shows the neutral-transpiler niche is still open — so the decision hinges on one question, not a feature tally:

If foundry goes multi-runtime →

Keep DNA — pivot it to the de-para layer

Author once, emit to agent-framework (proven), and to Bedrock / Vertex / OpenAI (cheap, ~20 LOC/target). Composition + eval-as-contract + tenant + no-.NET are real bonuses per-vendor formats lack. Cost: actually build the emitters and reckon with Oracle Agent Spec.

If foundry stays single-runtime →

Retire DNA → agent-framework-native + Langfuse

The de-para value is never realized, so its ceremony isn't worth it. Use AF's own declarative YAML and add Langfuse for the prompt versioning / A-B / true no-deploy that both DNA and AF lack today.

00 What agent-framework really does declaratively (not assumed)

Confirmed against the installed lib, not the marketing. agent-framework has a first-class declarative loader — and it is genuinely clean. It also carries a surprise tax.

It covers (natively, in YAML)

  • Persona / instruction — flat instructions: string
  • Model binding — model.{id, provider}
  • Tools — function / MCP / web-search / code-interpreter
  • Options + structured output — temperature, response schema
  • Loaded by AgentFactory.create_agent_from_yaml* → a live Agent

It does not cover — and one tax

  • No composition — no Soul reuse, no Guardrail-as-document; the ungrounded variant must copy the persona verbatim
  • No tenant / overlay — the PromptAgent model has no such field
  • No eval-as-contract — no prompt-invariant concept
  • .NET tax — the declarative loader hard-loads .NET 8 at import (pythonnet 3.0.5 + powerfx 0.0.34). It errored "must install .NET" on this box; only ran via DOTNET_ROLL_FORWARD=Major. Plain (imperative) agent-framework avoids .NET but is no longer declarative.

04 So — does DNA justify itself?

Straight answer, no hedging.

Not as it's used here. As a declarative prompt-manager for a single agent-framework backend, DNA does not pull its weight: agent-framework's own YAML loader is cleaner, already a dependency, and one fewer system to run. DNA's composition, eval-as-contract, tenant and no-.NET wins are genuine but too thin to justify the whole Genome/Kernel apparatus on their own.

Yes — but only if foundry commits to portability. DNA's one non-redundant reason to exist is as the neutral author-once → emit-to-each-runtime layer. The spike proves that works and is cheap (~20 LOC/target, byte-equal round-trip), and the 2026 landscape leaves the niche open (only Oracle Agent Spec, partially). That bet requires (1) foundry actually needing more than one runtime, (2) building the emitters, and (3) aligning with Oracle Agent Spec instead of a bespoke schema.

The decision reduces to one question: does foundry-assured target one agent runtime or many? One → retire DNA to agent-framework-native + Langfuse. Many → keep DNA and pivot it, deliberately, into the de-para layer it was almost accidentally already becoming.
