## Context

The self-wiki (`docs/wiki/foundry-helpdesk-*`) is generated by the **local Agent-Skills path**:
the coding agent reads the real code and writes a cited bundle, driven by the Microsoft
`wiki-architect` + `wiki-page-writer` skills we had vendored into
`apps/backend/app/knowledge/skills/`. Two things became clear while regenerating to v0.3.0:

1. **We already reuse Microsoft's wheel â€” but only 2 of ~11 skills.** Those two skills are part of
   the official [`microsoft/skills` â†’ `deep-wiki`](https://github.com/microsoft/skills/tree/main/.github/plugins/deep-wiki)
   plugin (MIT), which also ships `wiki-llms-txt`, `wiki-researcher`, `wiki-changelog`,
   `wiki-onboarding`, `wiki-qa`, `wiki-vitepress`, `wiki-agents-md`, `wiki-ado-convert` + `/deep-wiki:*`
   commands. We were **hand-rolling** things it already does (notably `llms.txt`, and the parallel
   multi-area orchestration).
2. **Keeping the wiki fresh is a solved pattern.** LangChain's [OpenWiki](https://github.com/langchain-ai/openwiki)
   (MIT, provider-agnostic incl. Anthropic) uses **git-diff â†’ a GitHub Action that opens a PR** to
   refresh docs as code changes â€” exactly the gap our `wiki-freshness` gate only *detects*.

The "don't reinvent" lesson repeats (cf. ADR-008/009/010/011): the generator and the freshness
automation are commodities; our value is the **assurance layer** on top.

## Decision

**Adopt the upstream `deep-wiki` plugin as the generation engine; keep our assurance layer; borrow
OpenWiki's freshness-automation *pattern*.**

- **Vendor the whole `deep-wiki` plugin** (pinned) at **`.github/skills/deep-wiki/`** â€” a
  GitHub-native skill-discovery location, so the **Copilot cloud agent, code review, CLI, and VS
  Code/JetBrains agent mode** surface it, and any teammate with GitHub Copilot runs the same
  generators. Claude Code / the local flow read it too. Provenance + pinned SHA in
  [`VENDORED.md`](../../.github/skills/deep-wiki/VENDORED.md). (Upstream is also installable directly:
  `/plugin install deep-wiki@skills`.)
- **Keep our layer as the moat â€” do NOT outsource it:** the ingest **bundle format**
  (`manifest.json` + `pages` + `llms.txt`), the **build-fidelity gate** (â‰¥80%), the **wiki-freshness
  gate**, the **Foundry KB ingest + ACL**, and the `--selfwiki` entrypoint.
- **Freshness â†’ regen, OpenWiki-style:** on drift, run the vendored `wiki-page-writer` skill via a
  coding agent in CI and open a PR. Shipped as a **manual (`workflow_dispatch`) template**
  ([`.github/workflows/wiki-regen.yml`](../../.github/workflows/wiki-regen.yml)) â€” it needs a model
  credential/agent, so it is **not** scheduled until that is wired.
- **DeepWiki (Cognition)** stays a **complement** for public human browsing
  (`deepwiki.com/ruinosus/foundry-assured`), not a source of truth: it's hosted, un-grounded to our
  KB, and has no ACL.

## Consequences

- **+** Stop maintaining a forked generator; inherit upstream fixes + the extra skills (`llms.txt`,
  onboarding, changelog, Q&A) for free.
- **+** GitHub-native discovery: no bespoke "how to run" â€” Copilot users invoke the skills directly.
- **+** A concrete path to close the freshness loop (detect â†’ auto-regenerate â†’ PR) instead of manual regen.
- **âˆ’** A vendored third-party tree to periodically re-pin (mitigated: `VENDORED.md` records the SHA;
