1 · O funcional — o que a pessoa vê

Pergunta → resposta com fontes → clico e vejo a fonte.

Nenhuma "alucinação": toda afirmação é ancorada num documento real da base de conhecimento, e o painel lateral lista as fontes citadas. Clicar numa fonte revela o documento de origem.

2 · A ideia-chave — dois caminhos, um por necessidade

Nem todo domínio precisa de ACL. Cada um usa o caminho certo.

O sistema é domain-swappable. Dois domínios "grounded" com necessidades diferentes → duas estratégias de recuperação, ambas rodando como o usuário via OBO e emitindo citações estruturadas para o mesmo painel.

📖

Selfwiki — acl=false audiência única

A wiki do próprio repositório é pública internamente — não há o que aparar por pessoa. Usa o caminho nativo da Microsoft: anexa a KB como MCP tool knowledge_base_retrieve inline na Responses API. A recuperação agentic (planejamento de query) dá ótima cobertura e devolve annotations url_citation nativas.

Caminho: Responses API + Foundry IQ MCP tool → citações nativas.

🛰️

Cockpit — acl=true por-usuário

A base do Cockpit tem documentos confidenciais. Cada pessoa só pode ver os seus. Usa recuperação + aparo do lado da app: uma busca direta como o usuário (onde a ACL funciona de verdade), que devolve só os documentos autorizados; a resposta é sintetizada apenas a partir desses documentos, e as fontes deles viram as citações.

Caminho: direct search (ACL trima) → síntese via Responses → citações dos docs autorizados.

Os dois entregam a mesma experiência (resposta fundamentada + painel de fontes clicável). A diferença é interna e existe por um motivo técnico concreto — o próximo slide.
3 · Por que dois caminhos — a descoberta técnica

A ACL por-usuário funciona na busca direta, mas não no caminho agentic.

Investigamos a fundo (e testamos ao vivo). O header de identidade do usuário só é honrado num dos caminhos de recuperação — e é isso que decide a arquitetura.

Caminho de recuperaçãoDá citações "bonitas"?Respeita ACL por-usuário?Evidência
Agentic knowledge_base_retrieve (MCP)Sim — url_citation nativas✗ Não trimaNossos testes (B via confidential) + issue Azure/azure-sdk-for-python #44454
Busca direta /docs/searchSem annotations — mas temos blob_url por doc✓ Trima (header x-ms-query-source-authorization)Nossos testes (A=7 / B=5) + Tutorial oficial ADLS ACL
Consequência: as citações nativas vêm do caminho agentic — que é justamente o que não respeita ACL. Então, para um domínio com ACL, não dá para usar o MCP tool: o modelo veria conteúdo confidencial. A solução correta é recuperar+aparar nós mesmos pela busca direta (onde a ACL funciona) e sintetizar só com o autorizado — fail-closed by design.
Investigamos até o caminho ADLS Gen2 + ACLs POSIX (ingestionPermissionOptions). Conclusão documentada: ele popula o metadado de grupo, mas não muda o fato de o retrieve agentic não trimar — a limitação está no retrieve path, não na ingestão. Por isso revertemos o ADLS e ficamos com o caminho de busca direta.
4 · Arquitetura — as três camadas

Frontend → Backend → Foundry na nuvem.

🖥️

Frontend · Next.js 15

CopilotKit renderiza o chat; MSAL faz o login Entra. O EvidencePanel escuta o stream AG-UI e mostra as fontes clicáveis. Fala com o backend via AG-UI sobre SSE.

⚙️

Backend · FastAPI (Python 3.12)

Endpoints /cockpit e /selfwiki chamam stream_grounded_agui. Resolve a identidade do usuário (OBO), recupera, sintetiza pela Responses API e re-emite eventos AG-UI.

☁️

Foundry + Azure AI Search

Microsoft Foundry hospeda o modelo (gpt-5-mini) e a Responses API. Azure AI Search (Foundry IQ) guarda a base + o campo de permissão groups que faz o trim por documento.

Tudo keyless: autenticação sempre via Entra ID / DefaultAzureCredential (nenhuma API key). Provisionado por azd + Bicep — um azd up do zero reproduz a stack inteira (é um repo-template).

6 · O outro caminho — selfwiki (nativo, sem ACL)

Quando não há ACL, usamos o padrão nativo da Microsoft.

A KB é anexada como MCP tool inline na Responses API. A Microsoft indica exatamente isso; a resposta já volta com as annotations url_citation — nós só as re-emitimos como o evento sources.

# build_responses_kwargs — o caminho acl=False
tools = [{
  "type": "mcp",
  "server_label": "knowledge-base",
  "server_url": f"{search}/knowledgebases/{kb}/mcp?api-version=2026-05-01-preview",
  "allowed_tools": ["knowledge_base_retrieve"],   # o único tool suportado
  "require_approval": "never",
  "authorization": search_token,                 # auth primária ao MCP server
}]
# instruções pedem citar como 【message_idx:search_idx†source_name】 (formato Microsoft)
# resposta traz References + url_citation → viram o evento CUSTOM sources
Esse caminho foi verificado ao vivo: o selfwiki respondeu fundamentado com 3 citações reais (url_citation com blob_urls de verdade). É o "jeito Microsoft" para grounding com citações — usado onde a ACL por-usuário não é requisito.

8 · A prova — ao vivo, A vs B

Mesma pergunta, acesso diferente.

Pergunta: "telemetria e observabilidade do Cockpit". O documento cockpit-mcp-telemetry é confidencial (só o grupo confidential lê). O usuário A pertence a ele; o B não.

👤 Usuário A — autorizado

7 documentos autorizados retornados.
Cita cockpit-mcp-telemetry-v1.2.0__page-1.md e …page-6.md.
Recebe a resposta completa sobre telemetria.

👤 Usuário B — só público

5 documentos autorizados retornados.
Nenhum deles é o de telemetria confidencial.
Não vê nem cita o conteúdo restrito. Fail-closed.

O modelo do B nunca recebe o trecho confidencial no contexto — o aparo acontece antes da síntese. Não é "esconder a citação"; é o conteúdo nunca entrar. Essa é a garantia de acesso do mecanismo (regra: acesso é dado, não lógica no código).
